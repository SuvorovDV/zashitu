# DECISIONS.md — принятые решения

Здесь фиксируем ключевые архитектурные и продуктовые решения с обоснованием. Перед тем как изменить что-то принципиальное — сначала сюда.

---

## Платформа

**Решение:** Telegram-бот (aiogram 3), не Mini App.
**Почему:** Mini App требует отдельный фронтенд, хостинг, HTTPS. Бот — это только Python-скрипт. Для MVP скорость важнее UX.
**Компромисс:** UX хуже (пошаговый диалог медленнее формы), но запускаемся быстрее.

---

## AI-модели

**Решение:** Sonnet 4.6 на базовом и стандарте, **Opus 4.7** на премиуме (обновлено 2026-04-17).
**Почему:** Sonnet достаточно хорош для генерации слайдов и в 5+ раз дешевле Opus. Opus 4.7 заметно лучше строит структуру, читает длинные источники и рисует SVG — это дифференциатор премиума.
**Цены:** Sonnet $3/$15 per MTok, Opus $15/$75 per MTok.
**Нюанс API:** Opus 4.7 **не принимает** `temperature` (deprecated). В `tasks.py` — `_messages_kwargs()` фильтрует параметр по имени модели, остальные вызовы (sonnet) идут с temperature как раньше.
**Исключение:** разметка речи при скачивании `.md` всегда идёт на `sonnet-4-6` независимо от тарифа (+ файловый кэш) — это простая задача, Opus там не нужен.

---

## Что не делаем

**Решение:** Не пишем дипломные и курсовые работы.
**Почему:** Антиплагиат, юридические риски, сложность (нужен RAG, chunking, рерайт). Масштаб задачи несоразмерен MVP.
**Что делаем вместо:** Берём уже написанную работу и делаем по ней презентацию.

## Конкурентная позиция

**Главный конкурент:** Gamma.app и аналогичные AI-presentation tools (Tome, Beautiful.ai, Canva AI).

**Ключевые отличия ZaShitu:**

1. **Source-grounded vs галлюцинации.** Gamma генерирует контент «из знаний» LLM — факты могут быть выдуманы. ZaShitu в режиме `source_grounded` работает **исключительно** с загруженным текстом. Каждый слайд → ссылка на страницу. Преподаватель на защите может проверить.

2. **Академический контекст.** Gamma не знает что такое ВКР, научный руководитель, кафедра, структура защиты. ZaShitu знает — FSM собирает именно эти данные и промты написаны под российский академический стандарт.

3. **Telegram-native.** Нет регистрации, нет браузера, нет подписки. Студент пишет боту там, где уже есть. Критично для мобильной аудитории.

4. **Доступность в РФ.** Gamma требует международную карту ($8–30/мес). Telegram Stars работают без Visa/Mastercard — для российских студентов это принципиально (санкции, отсутствие карт).

5. **Pay-per-generation.** Подписка неудобна студенту: нужна 1–2 раза в семестр. Gamma дерёт $8–30/мес. ZaShitu: 99–399 ⭐ один раз за нужную работу.

6. **Русский academic-first.** Gamma — English-first, русский вторичен. Всё у нас: промты, диалог, PDF — под русскоязычный контент.

**Чего мы НЕ делаем лучше Gamma:** дизайн, анимации, шаблоны, совместная работа. Не пытаемся конкурировать в этом — наша ниша другая.

## Режим source_grounded — главный режим

**Решение:** `source_grounded` — основной сценарий MVP, а не `no_template`.
**Почему:** Это главное отличие от конкурентов. Gamma делает презентации «из головы» — мы делаем по реальной работе студента с ссылками на страницы.
**Реализация:** Claude получает полный текст источника (PDF/DOCX) + промт с жёсткой инструкцией «работай только с этим текстом». Каждый слайд → поле `source: "стр. X"`.

## Хранение сессий

**Решение:** dict in-memory `{chat_id: FormData}` в MVP.
**Почему:** Redis — это инфраструктура, которую нужно поднимать и поддерживать. Для MVP с небольшим трафиком dict достаточно.
**Когда менять:** при горизонтальном масштабировании (несколько инстансов бота) или при необходимости сохранять сессии между перезапусками.

---

## Оплата

**Решение:** Telegram Stars через `send_invoice`.
**Почему:** Нативная для TG, не нужен внешний эквайринг для старта. Telegram берёт 30% — это известный факт, цены выставлены с учётом этого.
**Ограничение:** Нет нормального recurring billing — подписки через Stars на потом (этап 3+, веб-версия).

---

## Генерация асинхронно

**Решение:** `asyncio` + `await` для запроса к Claude API, `send_chat_action(UPLOAD_DOCUMENT)` пока ждём.
**Почему:** Генерация занимает 15–60 сек. Блокировать event loop нельзя — бот перестанет отвечать другим пользователям.

---

## Промты — модульная система

**Решение:** Промт собирается из нескольких файлов: CORE + тариф + режим + тип работы.
**Почему:** Комбинаций много (3 тарифа × 4 режима × 4 типа работы = 48). Модульная система позволяет менять одну ось, не переписывая всё.
**Самый короткий путь:** core + premium + source_grounded + vkr = 4 файла → уже работает для основного сценария.

---

## Формат вывода Claude

**Решение:** Claude возвращает JSON-массив слайдов, не текст.
**Почему:** JSON легко валидировать и передавать в pptx_builder. Текст потребовал бы парсинга.
**Схема:** `slide_schema_v1.json` — одна схема на все режимы. Поле `source` пустое для no_template, заполненное для source_grounded.

## Деплой: FirstVDS KVM + Docker

**Решение:** весь стек (бот + бэкенд + воркер + фронт + БД + Redis) живёт на FirstVDS KVM (Алматы, Казахстан) в одном `docker compose`. Бот polling напрямую в `api.telegram.org` без прокси.

**Почему FirstVDS, а не Yandex Cloud:**
- Yandex Cloud (РФ) требовал Cloudflare Worker-прокси к `api.telegram.org` (заблокирован в РФ)
- FirstVDS в Алматы — `api.telegram.org` доступен напрямую, прокси не нужен
- 2 ядра / 4 ГБ / 60 ГБ NVMe — с запасом для текущего стека

**Историческое:** ранее использовался Yandex Cloud VM `111.88.153.18` с Cloudflare Worker `tg-bot-proxy.erkobraxx.workers.dev`. Файлы `cloudflare-worker.js` и `wrangler.toml` остались в репо как артефакты, но больше не используются.

**FSM в MemoryStorage** (а не Redis): при рестарте контейнера пользователи на середине формы теряют состояние. Приемлемо для MVP; Redis — этап 2+.

## Монорепа (2026-04-15)

**Решение:** бот и `zashitu-web` (FastAPI-бэкенд + React-фронт) живут в одном репо `github.com/SuvorovDV/zashitu`. Структура — `bot/`, `backend/`, `frontend/`, `deploy/`.

**Почему:**
1. **Solo-dev, жёсткий контракт.** Бот — тонкий клиент бэкенда. Любое изменение эндпоинта требует синхронного изменения в двух местах; в монорепе это один атомарный PR, в двух репо — два коммита с риском рассинхрона.
2. **У `zashitu-web` не было `git init`.** Идеальный момент не заводить отдельную историю, а влить снапшотом в существующий репо.
3. **Меньше когнитивной нагрузки.** Один clone, один README, один `.env` на локальной машине, один CI.

**Что НЕ сделано этим решением (осознанно):**
- `contracts/shared.py` (общий источник правды для TIER/palette/work_type ID) не создан — у `bot/config.py` и `backend/config.py` разные поля в `TIERS`, и сразу извлекать общую часть значит рефакторить оба проекта. Отложено в PROGRESS.md «Следующий шаг».
- CI не настроен — сборка/тесты обоих процессов гоняются руками. Для solo-MVP приемлемо.
- `docker-compose` не объединён в один файл: корневой остался bot-only (чтобы не ломать текущий прод), а `deploy/docker-compose.prod.yml` содержит полный стек (bot + backend + worker + frontend + db + redis).

**Риски, принятые сознательно:**
- Правка `backend/` триггерит пересборку только бэкенда, но если у нас появится CI — он будет гонять всё при любом push. Для текущих объёмов десятки секунд, не проблема.
- React-фронт лежит рядом с Python-кодом — стандартная практика (Next.js + FastAPI монорепы), но визуально «перегружает» корень.

## Скелет слайдов выводится из утверждённой речи (2026-04-17)

**Решение:** при `include_speech=True` и утверждённой речи `_build_skeleton` делегирует построение скелета в `_derive_skeleton_from_speech` — отдельный Claude-pre-pass, который читает речь и возвращает titles+layouts под её реальные секции. Жёстко зашитый академический пул используется только как fallback (или когда речи нет).

**Почему:** в двух-этапном approval-флоу (сначала речь, потом слайды) пользователь утверждает речь по своей теме, а фиксированный пул («Введение → Цели и задачи → Обзор литературы → …») с ней часто не бьётся. Claude при заполнении контента либо натягивает реальный текст на несоответствующий title, либо оставляет слайд пустым. Плюс — маркировка речи маркерами `=== Слайд N ===` становится точной, потому что titles теперь ровно те же, что в речи.

**Цена:** +1 Claude-вызов на каждый `generate_slides_task` (≈5–10 сек, Opus 4.7). Приемлемо — приемка слайдов и так идёт ~60–90 сек.

**Risk:** если pre-pass упал или вернул `n != n_slides` — fallback на фиксированный пул (функция логирует warning, не прерывает основной флоу).

## SVG-декор через Claude + resvg, вместо крупных image-слайдов (2026-04-17)

**Решение:** иллюстрации в слайдах — не OpenAI `gpt-image-1`, а **SVG от Claude**, растеризованный через `@resvg/resvg-js` (Node, без системных Cairo-зависимостей). Крупные лейауты `image_full`/`image_side` убраны из rotation скелета — SVG теперь маленький декоративный акцент 1.5×1.5" в углу обычного слайда (`default`/`callout`), bullets занимают почти всю ширину.

**Почему:**
1. **Нет внешней платной зависимости.** OpenAI Images API требует отдельный аккаунт и международную карту. Claude уже подключён для всего остального.
2. **Palette-согласованность.** Claude видит `primary` и `accent` палитры слайда и рисует в них — получается единый стиль колоды. Stock-модель картинок выдаёт цвета вразнобой.
3. **Опус хорош в SVG-коде.** Проверено: Opus 4.7 выдаёт валидный SVG с осмысленной геометрией, без галлюцинаций текста/цифр.
4. **Академический жанр.** Крупная фотореалистичная картинка на каждом слайде отвлекает от содержания. Декоративная иконка в углу «оживляет» без перетягивания внимания.

**Как устроено:** `_images_via_claude_svg` передаёт Claude системный промпт с `viewBox="0 0 256 256"`, палитровыми хексами, запретом на `<text>` и реализм. Максимум 8 фигур. `svg_to_png.js` растеризует в 512px PNG. `pptxgen.js::cornerDecor` кладёт на слайд с `sizing: contain`.

**Fallback путь:** если `OPENAI_API_KEY` задан — используется OpenAI (старый код сохранён в `_images_via_openai`). Если нет — SVG через Claude. Если нет и ANTHROPIC_API_KEY — слайды без иллюстраций.

## Stats-карточки: 2×2 вместо 1×4 (2026-04-18)

**Решение:** в `pptxgen.js::statsSlide` при 4 показателях — сетка 2×2 с широкими плоскими карточками (4.5×1.7 дюйма) вместо узкого ряда 1×4 (2.15×2.3). При 3 — остаётся 1×3, но карточки шире. Левая primary-полоса `0.08"` по краю каждой карточки, квадратные углы вместо скруглённых, left-aligned 36/42pt number, 13pt label сразу под ним.

**Почему:** старый 1×4 layout с цифрами 48pt в узких карточках 2.15" страдал от автошринка PowerPoint на длинных значениях («14–16%», «40–50%»), а внутри карточки оставалось ~1.1" пустого места из-за valign center + маленькой подписи → выглядело недозагруженным. Бордер `CADCFC` на фоне `EEF3FB` был почти невидимым.

**Почему не меньше статов:** Claude-промпт просит 3–4 числа, оставляем всё. 2×2 при 4 — правильное решение 16:9 канваса.

**Цена:** две формы + доп. shape для accent-bar на карточку. Рендер по-прежнему <1 сек на весь deck. Коммит `8759218`.

## Chart-слайды: Calibri вместо Arial + data labels (2026-04-18)

**Решение:** в `chartSlide` передаются `catAxisLabelFontFace: null`, `valAxisLabelFontFace: null`, `dataLabelFontFace: null` — это подавляет default pptxgenjs-поведение (вшивание `<a:latin typeface="Arial"/>` в chart XML), и PowerPoint рендерит подписи в системном дефолте (Calibri), совпадая с остальным текстом колоды. Плюс `showValue: true` (кроме pie), `lineSize: 3`, marker 8pt, `chartColors: [pal.primary]` для 1-серии, только горизонтальные gridlines `#E5E7EB`.

**Почему:** pptxgen.js умолчально вшивает Arial в chart, в то время как addText-ы по всему deck рендерятся в Calibri — мелкий визуальный диссонанс на 9 графике. Одноцветная линия без data labels тоже просила проработки — без value-подписей цифры приходится читать по оси Y.

## Path C — закрытие контент-gap'а по сравнению с Leva (2026-04-18)

**Контекст:** сравнение с референс-деком «Leva» (HTML-презентация, тема «Рынок труда Кузбасса 2020–2025») показало три системных пробоя в нашем pptxgen-выходе: (1) USP «каждый тезис со ссылкой на страницу» не реализован — `source_ref` рендерился только на `default` layout, 29% покрытие; (2) Claude не вытаскивал именованные сущности (города, компании, ФИО) из источника, заменял их на «ключевые игроки», «депрессивные регионы»; (3) нет персонального голоса автора, хотя заказ его просит. Анализ от команды из 3 агентов (визуал / контент / tech-lead) — см. историю разговора от 2026-04-18.

**Решение:** сделали точечный pipeline-фикс (не переписывая рендер). Изменения:

1. **Pydantic schema** (`backend/generation/tasks.py`): `SlideContent.source_ref: Optional[str]` добавлено в модель.
2. **Fallback source_ref** расширен с `default` до всех content-layouts (`default/callout/two_col/stats/table/chart/image_side`). Если Claude не проставил — ставится «источник: загруженная работа».
3. **System prompt переписан** под редакторский voice с 5 разделами: критерии качества (NER pass / density floor / fact integrity / anti-patterns) → layout decision tree → citations (core USP) → JSON schema → 3 few-shot примера «эталонной плотности».
4. **pptxgen.js helper `sourceFooter(prs, slide, s)`** подключён ко всем content-layouts (кроме quote — у него attribution). Контент-area слайдов сжата с `5.25"→5.0"` чтобы зарезервировать 0.2" под футер.

**Валидация:** benchmark-fixture `bench/fixtures/smb_digital/` (курсовая ВШЭ «Цифровизация МСП 2020-2024»). Результат — `bench/REPORT.md`:
- `source_ref` coverage: 4/14 → 12/12 (100% на применимых слайдах)
- Named entities: ~10 → 18+ (СберБанк, Т-Банк, Минэкономразвития, Максим Решетников, Москва/СПб/Кемерово/Владивосток, «Код будущего», hh.ru и т.д.)
- Появился `quote` layout с авторской позицией от имени Морозовой А. — персональный голос, которого не было
- 6-мерная rubric: 17.5 → 20.5 / 30 (+17%), прирост в двух измерениях (визуальный сторителлинг +2, инфографика +0.5)
- Регрессий 0

**Бюджет:** ~$0.20 из $1.30 (sonnet-4-6 × 2 запуска по fixture). Осталось $1.10 на будущие итерации.

**Что не делали в этом спринте (отложено до Path A):** typography sweep (24pt→32pt title, eyebrow-mono 10pt), новые layouts `kpi_grid`/`chart_callouts`, chart-аннотации (peak markers), тематичные палитры под content-тему. Это визуальный gap, не контентный. Оценка — ещё +15-20% к rubric за отдельную неделю.

**NER-валидатор** (грепать каждую named-entity по source-тексту, реджектить галлюцинации) — НЕ реализован в этой итерации. На seeded speech не нашли проблем, но на реальных PDF нужен — backlog'ом.

**Bench-протокол:** `bench/README.md` описывает как прогонять fixture через worker-контейнер. Eval делает Claude Code session (не тратит API-ключ user'а). Budget per run ≈ $0.10 на sonnet.

## Enhance-режим: маркер «общее знание» фактически не работает (2026-04-18)

**Контекст:** в рамках B2 (`55d9f7f`) добавили `allow_enhance` — если юзер разрешает, Claude дополняет факты из общих знаний и помечает такие слайды `source_ref = «общее знание»`. USP сохраняется через прозрачную атрибуцию.

**Найдено при валидации** (4 конфига × 2 модели на sparse-fixture `bench/fixtures/sparse_ai`):

1. **На богатой речи** (smb_digital, 3.9k симв. с цифрами/таблицами) — оба Claude-а (sonnet-4-6 и opus-4-7) остаются в источнике, source_refs правильные, 0 enhance-пометок нужно. USP не ослабляется.
2. **На бедной речи** (sparse_ai, 866 симв. без цифр):
   - **Sonnet** дополняет СВОБОДНО («структурная безработица», «радиологи конкурируют с ИИ» — ни того ни другого в речи нет) и помечает «источник: загруженная работа» — то есть лжёт об атрибуции.
   - **Opus** enhance'ит почти ничего — строго парафразирует речь, помечает «Раздел X» — корректно, т.к. действительно перефраз.
3. **Маркер «общее знание» не использует ни одна модель.** Проверил raw JSON: оба выбирают безопасные fallback-и («источник: загруженная работа» или «Раздел X») вместо self-admission «я дополнил». Усиление промпта (явные правила + список признаков кандидатов на enhance-пометку) не изменило поведения. Paternalistic safety-bias Claude-моделей — не лечится промптингом разумной силы.

**Решения:**

1. **Маркер «общее знание» в системном промпте остаётся** — хотя бы документирует правило, плюс теоретически ловит редкие edge-случаи.
2. **UX-обещание «такие тезисы помечены общее знание» ослаблено** в копии Step 9 визарда — теперь пишем «best-effort атрибуция, модели склонны защищать вас от рисков и иногда не пользуются пометкой».
3. **Для академической честности (ВКР) рекомендовать Opus** — он меньше фантазирует на бедном источнике. На богатом — разницы почти нет.
4. **Self-audit pass** (второй Claude-проход, который именно помечает добавленное) — возможный backlog-item, но дорогой ($~0.10/заказ) и неочевидно что помогает (модель всё равно решает что считать дополнением). Пока не делаем.
5. **NER-валидатор** (грепать каждую named entity по source) становится важнее — это объективный checker, не полагается на self-admission Claude. Приоритет выше.

**Ship как есть.** Enhance-режим полезен для богатых источников (просто даёт Claude больше свободы формулировок) и относительно безопасен на бедных (Opus аккуратен, Sonnet дополняет но без явных галлюцинаций — добавляет обычные общеизвестные вещи, не fabricated numbers).

**Бюджет потрачен:** ~$1.80 из $31 на 5 конфигов валидации (sonnet × 4 standard + opus × 1 premium). Осталось $29.20.

## NER-валидатор: объективный бэкстоп для галлюцинаций (2026-04-18)

**Контекст:** после находки «Claude не пользуется маркером общее знание» нужен механизм, не полагающийся на self-admission модели. Валидатор из `ac9541dedf1b6df1d` agent-а — regex-экстрактор сущностей + стемминг-сверка с источником.

**Что реализовано** (`backend/generation/ner_validator.py`, ~270 строк):

- Regex-паттерны для 4 типов NE: топонимы (-ск/-цк/-ов/-ин/-ыш), геоадьективные фразы («область/край»), аббревиатуры (кириллица + латиница), числа-с-единицами (%/руб/₽/тыс/млн/млрд/трлн/чел/дн/мес), годы 19xx/20xx, ФИО (2-3 заглавных слова).
- Стоплисты: `_STOPLIST` (общеупотребимые Россия/Москва/ИИ/ООО), `_PHRASE_WORD_STOPLIST` (часть/раздел/глава/доклад — режет мета-конструкции).
- `_SYNONYMS` hand-curated: Кузбасс↔Кемеровская область, Т-Банк↔Тинькофф, МСП↔Малый и средний бизнес и т.п.
- Грубый стемминг под русскую морфологию (`_stem`): срезает типичные окончания падежей (-ая/-ой/-ую/-ого/...). Источник и entities стеммятся и сверяются — «Московская» и «Московскую» дают одинаковую базу.
- Валидация: для каждой NE из слайдов — проверка наличия в source через (1) phrase-level match, (2) синонимы, (3) token-level match. Числа — exact-match без стемминга.

**Интеграция** (`tasks.py::_generate_with_claude`):
- Вызов `validate_slides(contents, source)` после парсинга Claude-response.
- `source = speech + thesis + topic`.
- Warn-only: пишет `logger.warning` при находках, `logger.info clean` иначе. Retry/reject **не** делается — копим сигнал, прежде чем принимать решения.
- Результат `hallucinated: list[{entity, slide_idx, layout}]` пишется в `prompt_record['hallucinated_entities']` → сохраняется в `orders.generation_prompt` → доступен для аналитики и post-mortem.
- Behind env flag `NER_VALIDATE_ENABLED` (default True, можно выключить).

**Валидация на корпусе:**

| Дек | Ожидание | Результат |
|---|---|---|
| Path C baseline (rich, strict) | 0 FP | 0 ✓ |
| Path C after (rich, strict) | 0 FP | 0 ✓ |
| Sparse Sonnet (enhance) | детект phrase-level дополнений | 0 (limitation — entity-level миссит парафраз) |
| Sparse Opus (enhance) | консервативный пересказ | 0 ✓ |
| Синтетика с 7 подложенными NE | ≥5 поймать | 4/7 (СУЭК, 2028, Владимир Путин, «Цитата Владимира»); миссит «Прокопьевске» (склонённый падеж), «14,5 %» (edge regex), «Распадская» (adj-form), «3 000» (без единицы) |

**Ограничения:**
1. Phrase-level галлюцинации (Claude добавил «структурная безработица среди специалистов среднего звена» без конкретных named entities) — не детектятся. Нужна семантика, regex не хватит. Будет решаться позже другим механизмом (embedding-sim или second-Claude-audit-pass).
2. Склонённые топонимы в именительном падеже source'а ↔ оригинальный падеж slide'а — частично решается через стемминг, но edge cases остаются.
3. Stoplists handlabel-maintained — придётся пополнять по мере встречи false-positives в проде.

**Ship strategy:** observability-only сейчас. Через ~20 реальных заказов смотрю распределение `hallucinated_entities`. Если стабильно 0-2 на дек — принимаю порог 3+ как сигнал для авторетрая со strict-промптом. Если >5 — тюнинг регексов/стоп-листов.

**Бюджет потрачен:** ещё $0.60 на prod-smoke-тест NER пайплайна. Всего $2.40 из $31. Осталось $28.60.
