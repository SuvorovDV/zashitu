# DECISIONS.md — принятые решения

Здесь фиксируем ключевые архитектурные и продуктовые решения с обоснованием. Перед тем как изменить что-то принципиальное — сначала сюда.

---

## Платформа

**Решение:** Telegram-бот (aiogram 3), не Mini App.
**Почему:** Mini App требует отдельный фронтенд, хостинг, HTTPS. Бот — это только Python-скрипт. Для MVP скорость важнее UX.
**Компромисс:** UX хуже (пошаговый диалог медленнее формы), но запускаемся быстрее.

---

## AI-модели

**Решение:** Sonnet 4.6 на базовом и стандарте, Opus 4.6 только на премиуме.
**Почему:** Sonnet достаточно хорош для генерации слайдов и в 5+ раз дешевле Opus. Opus — дифференциатор премиума, не основа.
**Цены:** Sonnet $3/$15 per MTok, Opus $5/$25 per MTok.

---

## Что не делаем

**Решение:** Не пишем дипломные и курсовые работы.
**Почему:** Антиплагиат, юридические риски, сложность (нужен RAG, chunking, рерайт). Масштаб задачи несоразмерен MVP.
**Что делаем вместо:** Берём уже написанную работу и делаем по ней презентацию.

---

## Конкурентная позиция

**Главный конкурент:** Gamma.app и аналогичные AI-presentation tools (Tome, Beautiful.ai, Canva AI).

**Ключевые отличия ZaShitu:**

1. **Source-grounded vs галлюцинации.** Gamma генерирует контент «из знаний» LLM — факты могут быть выдуманы. ZaShitu в режиме `source_grounded` работает **исключительно** с загруженным текстом. Каждый слайд → ссылка на страницу. Преподаватель на защите может проверить.

2. **Академический контекст.** Gamma не знает что такое ВКР, научный руководитель, кафедра, структура защиты. ZaShitu знает — FSM собирает именно эти данные и промты написаны под российский академический стандарт.

3. **Telegram-native.** Нет регистрации, нет браузера, нет подписки. Студент пишет боту там, где уже есть. Критично для мобильной аудитории.

4. **Доступность в РФ.** Gamma требует международную карту ($8–30/мес). Telegram Stars работают без Visa/Mastercard — для российских студентов это принципиально (санкции, отсутствие карт).

5. **Pay-per-generation.** Подписка неудобна студенту: нужна 1–2 раза в семестр. Gamma дерёт $8–30/мес. ZaShitu: 99–399 ⭐ один раз за нужную работу.

6. **Русский academic-first.** Gamma — English-first, русский вторичен. Всё у нас: промты, диалог, PDF — под русскоязычный контент.

**Чего мы НЕ делаем лучше Gamma:** дизайн, анимации, шаблоны, совместная работа. Не пытаемся конкурировать в этом — наша ниша другая.

---

## Режим source_grounded — главный режим

**Решение:** `source_grounded` — основной сценарий MVP, а не `no_template`.
**Почему:** Это главное отличие от конкурентов. Gamma делает презентации «из головы» — мы делаем по реальной работе студента с ссылками на страницы.
**Реализация:** Claude получает полный текст источника (PDF/DOCX) + промт с жёсткой инструкцией «работай только с этим текстом». Каждый слайд → поле `source: "стр. X"`.

---

## Хранение сессий

**Решение:** dict in-memory `{chat_id: FormData}` в MVP.
**Почему:** Redis — это инфраструктура, которую нужно поднимать и поддерживать. Для MVP с небольшим трафиком dict достаточно.
**Когда менять:** при горизонтальном масштабировании (несколько инстансов бота) или при необходимости сохранять сессии между перезапусками.

---

## Оплата

**Решение:** Telegram Stars через `send_invoice`.
**Почему:** Нативная для TG, не нужен внешний эквайринг для старта. Telegram берёт 30% — это известный факт, цены выставлены с учётом этого.
**Ограничение:** Нет нормального recurring billing — подписки через Stars на потом (этап 3+, веб-версия).

---

## Генерация асинхронно

**Решение:** `asyncio` + `await` для запроса к Claude API, `send_chat_action(UPLOAD_DOCUMENT)` пока ждём.
**Почему:** Генерация занимает 15–60 сек. Блокировать event loop нельзя — бот перестанет отвечать другим пользователям.

---

## Промты — модульная система

**Решение:** Промт собирается из нескольких файлов: CORE + тариф + режим + тип работы.
**Почему:** Комбинаций много (3 тарифа × 4 режима × 4 типа работы = 48). Модульная система позволяет менять одну ось, не переписывая всё.
**Самый короткий путь:** core + premium + source_grounded + vkr = 4 файла → уже работает для основного сценария.

---

## Формат вывода Claude

**Решение:** Claude возвращает JSON-массив слайдов, не текст.
**Почему:** JSON легко валидировать и передавать в pptx_builder. Текст потребовал бы парсинга.
**Схема:** `slide_schema_v1.json` — одна схема на все режимы. Поле `source` пустое для no_template, заполненное для source_grounded.

## Деплой: FirstVDS KVM + Docker

**Решение:** весь стек (бот + бэкенд + воркер + фронт + БД + Redis) живёт на FirstVDS KVM (Алматы, Казахстан) в одном `docker compose`. Бот polling напрямую в `api.telegram.org` без прокси.

**Почему FirstVDS, а не Yandex Cloud:**
- Yandex Cloud (РФ) требовал Cloudflare Worker-прокси к `api.telegram.org` (заблокирован в РФ)
- FirstVDS в Алматы — `api.telegram.org` доступен напрямую, прокси не нужен
- 2 ядра / 4 ГБ / 60 ГБ NVMe — с запасом для текущего стека

**Историческое:** ранее использовался Yandex Cloud VM `111.88.153.18` с Cloudflare Worker `tg-bot-proxy.erkobraxx.workers.dev`. Файлы `cloudflare-worker.js` и `wrangler.toml` остались в репо как артефакты, но больше не используются.

**FSM в MemoryStorage** (а не Redis): при рестарте контейнера пользователи на середине формы теряют состояние. Приемлемо для MVP; Redis — этап 2+.

## Монорепа (2026-04-15)

**Решение:** бот и `zashitu-web` (FastAPI-бэкенд + React-фронт) живут в одном репо `github.com/SuvorovDV/zashitu`. Структура — `bot/`, `backend/`, `frontend/`, `deploy/`.

**Почему:**
1. **Solo-dev, жёсткий контракт.** Бот — тонкий клиент бэкенда. Любое изменение эндпоинта требует синхронного изменения в двух местах; в монорепе это один атомарный PR, в двух репо — два коммита с риском рассинхрона.
2. **У `zashitu-web` не было `git init`.** Идеальный момент не заводить отдельную историю, а влить снапшотом в существующий репо.
3. **Меньше когнитивной нагрузки.** Один clone, один README, один `.env` на локальной машине, один CI.

**Что НЕ сделано этим решением (осознанно):**
- `contracts/shared.py` (общий источник правды для TIER/palette/work_type ID) не создан — у `bot/config.py` и `backend/config.py` разные поля в `TIERS`, и сразу извлекать общую часть значит рефакторить оба проекта. Отложено в PROGRESS.md «Следующий шаг».
- CI не настроен — сборка/тесты обоих процессов гоняются руками. Для solo-MVP приемлемо.
- `docker-compose` не объединён в один файл: корневой остался bot-only (чтобы не ломать текущий прод), а `deploy/docker-compose.prod.yml` содержит полный стек (bot + backend + worker + frontend + db + redis).

**Риски, принятые сознательно:**
- Правка `backend/` триггерит пересборку только бэкенда, но если у нас появится CI — он будет гонять всё при любом push. Для текущих объёмов десятки секунд, не проблема.
- React-фронт лежит рядом с Python-кодом — стандартная практика (Next.js + FastAPI монорепы), но визуально «перегружает» корень.

## Переход прода на монорепу в одном compose (2026-04-15)

**Решение:** прод запускается из `~/zashitu/deploy/docker-compose.prod.yml` с `-p deploy`. Все 6 сервисов в одной сети Docker, один `deploy/.env.prod`.

**Что осталось хрупким:**
- `DEV_MODE=True` на проде — потому что Stripe не подключён, а `/dev/simulate_payment` всё ещё нужен для ручных тестов. Перед публичным запуском — выключить.
- Нет git на VM. Обновления идут через `scp tarball + tar -xzf`. В скрипте обновления нужно сохранять `deploy/.env.prod`.
